# ADR 0022: Package by feature for application code

- **Status**: Proposed
- **Date**: 2026-05-15
- **Deciders**: Repo platform
- **Tags**: layout, conventions, frontend, backend

## Context

Two patterns exist for organising non-trivial application code:

- **Package by layer** — top-level folders by *kind*: `components/`, `queries/`, `schemas/`, `services/`, `controllers/`, `models/`. Each folder pools every feature's files together.
- **Package by feature** — top-level folders by *domain*: `inventory/`, `billing/`, `auth/`. Each folder owns the full vertical slice (resolver + service + types + tests; or queries + schemas + components). Cross-domain primitives go into a small set of type-folders.

Also called **vertical slice architecture** when the slice cuts through multiple layers.

NestJS feature modules ([ADR 0006](0006-nestjs-backend.md)) already encode package-by-feature on the backend. Frontend frameworks ([ADR 0007](0007-frontend-frameworks.md)) don't enforce either pattern; without a convention, they drift into package-by-layer because that's what most tutorials show.

This ADR locks in one convention so every app scaffolded from base-app starts the right way.

### Why package by feature

1. **Things that change together live together.** Adding a column to a list touches the query, the schema, and the component. Layered = three folders; feature = one.
2. **Dead code is obvious.** Deleting a feature is one `rm -rf`, not a five-folder audit. Knip's job sharpens.
3. **The codebase reads like the product.** `inventory/`, `billing/`, `auth/` tells a new contributor what the app *does*. `components/` tells them what the framework supports.
4. **Boundaries are visible.** Cross-feature imports (`lib/inventory/` → `lib/billing/`) show in the import path. In a layered layout, every import looks the same; coupling is invisible until it bites.
5. **Refactor and rename are local.** Renaming a domain is one folder rename plus import-path updates, not a diff scattered across the tree.

## Decision

Adopt **package by feature** as the default file-organisation convention for application code.

### Scope

Applies to:
- Backend services under `apps/<product>/<service>/src/` (NestJS apps, workers, CLIs).
- Frontend application code under `apps/<product>/<service>/src/lib/` (SvelteKit) or `apps/<product>/<service>/src/` (React + Vite).

Does **not** apply to:
- Static / content-driven sites with no app domain (e.g. Astro marketing sites). A pages-only site has no feature to package by; type-folders are fine.
- `shared/<lib>/` packages, which are themselves the unit of feature ownership.
- Generated output (`lib/gql/`, codegen-only directories).
- SvelteKit `routes/` or Next-style file-routing trees, which are already URL-organised (and URL usually tracks domain).

### Rules

1. **Domain code lives in a feature folder.** Every file whose primary reason to change is a single domain belongs in `<domain>/`. Folder name is the domain noun, lowercase.
2. **Co-locate the full slice.** Tests, types, queries, schemas, components for one domain sit in the same folder. No sibling `tests/` / `types/` / `schemas/` folders elsewhere for files that belong to one domain.
3. **Files are flat per feature** until the count crosses ~10. `inventory.resolver.ts` next to `inventory.service.ts` — no `resolvers/` subfolder.
4. **Cross-domain primitives go in a type-folder.** Generic UI (`ui/`), codegen output (`gql/`), design tokens (`styles/`), infrastructure clients (`database/`). These have no domain to belong to.
5. **A file moves out of its feature folder only when a second feature imports it.** Until then, "shared" is hypothetical. Premature extraction hides the domain that actually owns the file.
6. **App-shell chrome is a feature.** Top bar, left rail, command palette — name it `shell/`, not `layout/` or `chrome/`.

### Boundary heuristics

- Used by **one** route family today, no clear plan for re-use → feature folder.
- Used by **two or more** route families across distinct domains → type-folder.
- Generated by tooling → type-folder.
- Generic primitive (button, tooltip, avatar) with no domain meaning → type-folder (`ui/`).

When in doubt, start in the feature folder. Moving out is cheap; moving in is awkward because callers that should not have known about the file already do.

## Consequences

### Positive

- Backend and frontend share one convention; contributors move between them without re-learning layout.
- New features land as a single folder add. No five-folder coordinated diff.
- Whole-feature deletes are one `rm -rf` — knip and code review both get simpler.
- Cross-feature coupling surfaces in the diff as visible import paths.

### Negative

- "Where does this file go" requires a judgement call. The heuristics above cover most cases; borderline files (e.g. a formatter used by two-and-a-half domains) need an explicit decision.
- Slightly deeper paths: `lib/inventory/queries.ts` instead of `lib/queries/inventory.ts`. Rough wash on import legibility; clear win on co-location.

### Neutral

- Plays well with shadcn-style component libraries (`lib/ui/components/`) — those install into the type-folder this ADR sanctions, so no conflict.

## Alternatives considered

### 1. Package by layer

- **Pro**: matches most framework tutorials; familiar.
- **Con**: scales badly past two domains; spreads each feature across the tree; backend already rejected it; no diff-locality for feature changes. Rejected.

### 2. Hybrid — feature folders inside `components/`, type folders for everything else

- **Pro**: less disruption for projects already on package-by-layer.
- **Con**: feature still lives in three places — solves nothing. Rejected.

### 3. Per-route co-location only (SvelteKit's `_components` convention)

- **Pro**: tighter co-location for route-private files.
- **Con**: only works for files used by exactly one route. Multi-route domain code needs a reusable home; `lib/<feature>/` is that home. Route-private files can still live in `routes/<route>/` when they really are private.

## Implementation plan

base-app ships no application code today, so no migration. Future apps follow the rule from creation:

1. When `nx g` scaffolds a new app, the generator template starts with empty `ui/` (type-folder) rather than empty `components/` (layered). Tracked separately as a generator update.
2. When an app from a downstream project arrives layered, the first non-trivial feature addition is the trigger to convert. Per-app `CLAUDE.md` should document its layout state.
