# ADR 0031: Structured logging contract

## Context and Problem Statement

Without a shared log shape, agents and humans can't grep
across services, trace IDs don't correlate, and switching
backend requires code changes per fork. This ADR fixes the
shape, the emission path, and local capture. Production
routing is in [ADR 0032](0032-runtime-observability.md).

## Decision Outcome

### Logger — pino 10 via `nestjs-pino`

`pino` + `nestjs-pino` register as the NestJS logger;
framework logs, `HttpException`s, and application code all
flow through one pino instance. Versions pinned exactly in
each service's `package.json` per repo convention.

### Stdout-only — no transports in app code

Pino writes JSON to stdout. **No transport packages in
`dependencies`**: `pino-cloudwatch`, `pino-loki`,
`winston-cloudwatch`, `pino-sentry`, etc. are banned. Routing
to a backend (CloudWatch, Loki, New Relic, …) is the
infrastructure's job — see [ADR 0032](0032-runtime-observability.md).

Biome's `noRestrictedImports` rule enforces this in
`biome.jsonc`. CI fails on a new transport import.

### Log shape — one JSON object per line

Required fields on every log line:

| Field | Source | Why |
|---|---|---|
| `level` (numeric) | pino default | Standard pino contract |
| `severity` (string) | custom formatter | Human/agent-readable mirror of `level` |
| `time` (epoch ms) | pino default | Sort, range-filter |
| `msg` | pino default | Free-text payload |
| `service.name`, `service.version` | resource attrs | Multi-service correlation |
| `trace_id`, `span_id`, `trace_flags` | `@opentelemetry/instrumentation-pino` | Cross-service correlation |
| `err.type`, `err.message`, `err.stack` | `pino.stdSerializers.err` | OTel exception semantics |
| `event` (string, app-supplied) | code | Stable key for grouping (e.g. `http.request.completed`) |

Base config lives in `shared/logger/` — a `pino.LoggerOptions`
exporting the `level`/`severity` formatter, `err` serialiser,
and `LOG_LEVEL` env binding. Services import and extend.

### Local capture — tee raw JSON to `.ai-wip/logs/`

Default `pnpm dev` per service pipes through `tee`:

```
node --import ./dist/instrumentation.mjs dist/main.js \
  | tee .ai-wip/logs/<product>-<service>.log \
  | pino-pretty
```

- **Raw JSON** lands in `.ai-wip/logs/<product>-<service>.log` —
  agents grep/jq it across tool-call boundaries.
- **Pretty output** to terminal for human eyes.
- `.ai-wip/` is gitignored (CLAUDE.md "Capture output for review").

The wiring lives in each service's `package.json` `dev` script,
generated by the NX generator — not duplicated by hand.

### `pnpm watch-logs` — repo-root convenience

`tail -f .ai-wip/logs/*.log | pino-pretty`. Humans run it from
any directory; agents read the files directly with `rg`/`jq`.

## Consequences

### Positive

- **One log shape across services** — agents and humans grep
  the same fields everywhere.
- **Backend swap is an infra change** — app code never knows
  where logs go.
- **Agents have first-class log access** in `.ai-wip/logs/` —
  survives tool-call boundaries, structured, queryable.
- **Trace correlation by construction** —
  `@opentelemetry/instrumentation-pino` injects `trace_id`
  into every line inside a span.

### Negative

- **`tee` adds a buffer layer** — a crash mid-flush can lose
  the last few lines from the file. Acceptable for local dev;
  prod runs without `tee`.
- **Two streams locally** (file + pretty terminal) — humans
  must understand the file is the source of truth.

### Neutral

- **Frontend logging** out of scope — React errors via the
  error-layer SOP in [ADR 0032](0032-runtime-observability.md).
- **Worker / batch services** follow the same contract.

## Alternatives considered

1. **Winston** — ~5–8× CPU per line and harder to integrate
   with OTel's pino instrumentation. Rejected.
2. **`pino-pretty` as an in-process transport** (incl.
   `NODE_ENV`-switched) — couples prod and dev paths; dev/prod
   diverge in serialised shape. Pretty-print stays
   out-of-process via `| pino-pretty`.
3. **Timestamped log file per `pnpm dev` invocation** —
   agents need a stable path to re-read across tool calls.
